# Лабораторная работа №2.2

## Введение
Данный проект представляет собой расширенную реализацию системы для работы с задачами. В отличие от предыдущей версии (ЛР 2.1), здесь основное внимание уделено внутренней модели задачи с использованием **дескрипторов (descriptors)** для валидации и управления атрибутами. Проект демонстрирует углубленное применение принципов SOLID, работу с исключениями, продвинутое логирование, а также разделение ответственности между компонентами системы.

## Описание компонентов

### models (Модели данных)
- `descriptors.py`: Содержит классы-дескрипторы для управления атрибутами модели `Task`.
  - `ValidatedString`: Data descriptor для строк с проверкой длины.
  - `PositiveInteger`: Data descriptor для целых чисел в заданном диапазоне.
  - `PriorityDescriptor`: Специализированный дескриптор для приоритета, возвращающий словарь со значением и названием уровня (Low, Medium, High, Critical).
  - `StatusDescriptor`: Non-data descriptor для статуса с проверкой допустимых переходов (pending → running → completed/canceled).
  - `TimestampDescriptor`: Data descriptor для временных меток с автоматической установкой.
  - `DataDescriptor`: Non-data descriptor для доступа к полезной нагрузке (`payload`).

- `exceptions.py`: Определяет иерархию исключений для валидации.
  - `TaskValidationExcept`: Базовое исключение.
  - `InvalidPriorityError`, `InvalidStatusError`, `InvalidDescriptionError`, `InvalidIDError`: Специализированные исключения.

- `task.py`: Содержит класс `Task`, который использует дескрипторы для атрибутов.
  - **Атрибуты**: `id`, `description`, `priority`, `created_at`, `status`, `payload`.
  - **Вычисляемые свойства**: `is_ready`, `age`, `age_formatted`, `priority_name`, `is_completed`.
  - **Методы**: `upd_status()` для безопасного изменения статуса, `to_dict()` и `from_dict()` для сериализации.

### src (Исходный код приложения) – наследие ЛР1
- `contracts.py`: Содержит базовые контракты системы (датакласс `Task` и протокол `TaskSource`), а также функцию `check_task_source` для проверки соответствия протоколу.
- `logger_config.py`: Настраивает логирование в консоль и файл с динамическим именем.
- `sources.py`: Реализации источников задач (`FileTaskSource`, `GeneratorTaskSource`, `APITaskSource`).
- `processor.py`: Класс `TaskProcessor` для управления источниками и обработки задач.
- `main.py`: Точка входа, демонстрирующая работу источников (ЛР1), модели с дескрипторами (ЛР2) и их интеграцию.

### tests (Тесты)
- `test_contracts.py`: Проверка протокола `TaskSource` и функции проверки.
- `test_sources.py`: Тестирование каждого источника задач.
- `test_processor.py`: Юнит-тесты для класса `TaskProcessor`.
- `test_integration.py`: Интеграционные тесты, проверяющие совместную работу всех компонентов, включая создание файлов и реальное взаимодействие.

## Демонстрация возможностей

### 1. Работа с дескрипторами (ЛР2)
- **Валидация**: При создании задачи проверяется длина `id` и `description`, значение `priority` (1-4) и допустимость начального `status`.
- **Управление статусом**: Метод `upd_status()` контролирует переходы между статусами, запрещая некорректные изменения (например, из `completed` в `running`).
- **Автоматические поля**: `created_at` устанавливается автоматически при создании задачи.
- **Вычисляемые свойства**: `age_formatted` возвращает удобочитаемый возраст задачи, `priority_name` — название уровня приоритета.

### 2. Интеграция с источниками (ЛР1)
- `TaskProcessor` может принимать любые источники, соответствующие протоколу `TaskSource`, включая пользовательские, которые генерируют задачи модели `Task` из `models.task`.

### 3. Логирование
- Все операции (создание задач, изменение статуса, валидация, работа источников) логируются в консоль и в файл с меткой времени.

## Требования к коду и инструментарий
При разработке данного проекта соблюдались следующие правила и использовались инструменты:

- **Стиль кода**: Код написан в соответствии с PEP8.
  - `snake_case` для имен переменных, функций и модулей.
  - `PascalCase` для имен классов.
  - Отступы: 1 шаг = 4 пробела.
- **Документирование**: Код содержит `docstring` для классов и функций, а также аннотации типов (`type hints`).
- **Логирование**: Используется модуль `logging` с выводом в консоль и файл.
- **Дескрипторы**: Для валидации и управления атрибутами применены data и non-data дескрипторы.
- **Обработка исключений**: Создана иерархия собственных исключений для различных ошибок валидации.
- **Тестирование**: Используются `unittest` и `pytest` для покрытия функциональности.
- **Контрактное программирование**: Протокол `TaskSource` обеспечивает гибкость и расширяемость системы.
