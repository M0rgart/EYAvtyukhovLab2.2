# Лабораторная работа №2.1

## Введение
Данный проект представляет собой реализацию системы для получения и обработки задач (Task) из различных
источников данных. В основе архитектуры лежит использование протоколов Python для определения контракта
источника задач, что обеспечивает гибкость и расширяемость системы. Проект демонстрирует принципы SOLID,
работу с логгированием, обработку исключений и современный подход к организации кода на Python.

## Описание компонентов

### src (Исходный код)
   - `contracts.py`: Содержит базовые "контракты" системы.
     - class Task: Датакласс, представляющий собой единицу работы. Содержит id и payload (полезную нагрузку).
     - class TaskSource(Protocol): Протокол, определяющий интерфейс для любого источника задач. Класс, который хочет быть источником, должен реализовывать метод get_tasks() -> list[Task].
     - check_task_source(task: any) -> bool: Функция для проверки, соответствует ли переданный объект протоколу TaskSource, с подробным логгированием результата проверки.
   - `logger_config.py`: Отвечает за настройку системы логгирования.
     - setup_logfing(level=logging.INFO): Функция, которая настраивает вывод логов одновременно в консоль и в файл. Имя файла генерируется динамически на основе текущей даты и времени (task_sources_YYYYMMDD-HHMMSS.log).
   - `sources.py`: Содержит конкретные реализации источников задач, соответствующих протоколу TaskSource.
     - class FileTaskSource: Читает задачи из JSON-файла. Если в записи файла нет поля id, генерирует его автоматически.
     - class GeneratorTaskSource: Генерирует заданное количество тестовых задач с случайным наполнением.
     - class APITaskSource: Имитирует получение задач из внешнего API (содержит мок-данные).
   - `processor.py`: Содержит класс TaskProcessor, который управляет источниками и обрабатывает задачи.
     - add_source(source: TaskSource) -> bool: Добавляет источник в пул обработки, предварительно проверяя его соответствие контракту через check_task_source.
     - process_all() -> List[Task]: Последовательно проходит по всем добавленным источникам, собирает из них задачи и возвращает общий список. Ошибки при получении задач из одного источника не прерывают обработку остальных.
     - get_sorce_count() -> int: Возвращает количество добавленных источников.
   - `main.py`: Точка входа в программу.
     - Функция create_file() создает пример JSON-файла (file1.json) с данными для демонстрации.
     - Функция main() демонстрирует полный цикл работы программы: настройка логгирования, создание тестового файла, инициализация процессора, добавление всех трех типов источников и запуск обработки задач с выводом результатов.

### tests (Тесты)
   - `test_contracts.py`: Тестирование протокола и функции проверки.
   - `test_sources.py`: Проверка каждого источника задач (FileTaskSource, GeneratorTaskSource, APITaskSource) изолированно. 
   - `test_processor.py`: Юнит-тесты для класса TaskProcessor.
   - `test_integration.py`: Интеграционные тесты, проверяющие совместную работу всех компонентов.

## Требования к коду и инструментарий
При разработке данного проекта соблюдались следующие правила и использовались инструменты:

Код написан в соответствии с PEP8.

Snake_case для имен переменных, функций и модулей (например, get_tasks, logger_config).

PascalCase для имен классов (например, FileTaskSource, TaskProcessor).

Осмысленные имена сущностей.

Отступы: 1 шаг = 4 пробела.

Документирование: Код содержит docstring для классов и функций, а также аннотации типов (type hints) для аргументов и возвращаемых значений.

Логирование: Вся работа приложения протоколируется с помощью модуля logging. Настроена запись как в консоль, так и в файл с уникальным именем.

Контрактное программирование: Использование Protocol из модуля typing позволяет явно объявить интерфейс TaskSource и проверять его соблюдение во время выполнения программы.

Менеджер зависимостей: Для управления пакетами используется uv, что подтверждается наличием файлов uv.lock и pyproject.toml. Для обратной совместимости также добавлен requirements.txt.
